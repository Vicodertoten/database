- Phase 0 benchmark is locked to one owner difficulty policy: `mixed`

  - `start_round_or_session -> get_question -> submit_answer`

## Benchmark protocol (owner primary)

- 3 consecutive owner runs
- same commands, same segment, same snapshot policy
- one comparable corpus/snapshot per run
- no KPI formula change across runs

Formulas:

- `compile_success_ratio_segment = builds(question_count_built == question_count_requested) / total_builds_attempted`
- `distractor_diversity_segment = unique_directed_pairs(target_canonical_taxon_id, distractor_canonical_taxon_id) / total_distractor_slots`

## Consumer runtime validation protocol

- measure `latency_e2e_segment_p95` on consumer side in frozen environment
- explicit warmup before measurements
- minimum 30 runtime requests per run for exploitable p95
- compare against prototype baseline using the same measurement protocol

## Optional live stability check (separate)

- run live iNaturalist smoke separately
- report as `stability_check_live`
- never use this track as the primary comparative benchmark signal

## Go/No-Go rule

- Go: all 4 criteria reached on 3 consecutive comparable runs
- No-Go: any threshold missed or comparability not demonstrated
