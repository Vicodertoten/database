## Official fixture strategy (locked for INT-003)

### Explicit decision (2026-04-15)

- Fixtures are **minimal valid payloads**, not exhaustive payloads.
- `playable_corpus.sample.json` is a **reduced subset** of a real `playable_corpus.v1` output.
- `pack_compiled.sample.json` is a **real compile output** with `question_count=1`.
- `pack_materialization.sample.json` is a **real materialization output** with `question_count=1`.
- `pack_materialization.sample.json` purpose is locked to `daily_challenge`.
- Target usage is **local consumption by `runtime-app`**, not product-exhaustive coverage.

Direct answers to the decision points:

- complete or minimal fixtures: minimal valid fixtures
- coherence between files: same real build/materialization lineage and same 4 playable item IDs (1 target + 3 distractors) across all files
- purpose for materialization: `daily_challenge`, to cover the purpose branch with required `ttl_hours` + `expires_at`
- target size for local use: `playable_corpus.sample.json` with 4 items; `pack_compiled.sample.json` with 1 question; `pack_materialization.sample.json` with 1 question

### Scope

Only these runtime surfaces are eligible for official fixtures:

- `playable_corpus.v1`
- `pack.compiled.v1`
- `pack.materialization.v1`

`export.bundle.v4` is explicitly forbidden as a runtime fixture source.

### Ownership and publication rule

- Official fixture truth is published first in `database`.
- `runtime-app` only mirrors and consumes published fixtures.
- Any fixture schema interpretation conflict is resolved in `database` first.

### Fixture provenance rule

Each official fixture must be derived from real owner-side artifact generation paths (inspect/pack/materialization outputs), not handcrafted consumer payloads.

### Fixture set model

Each official minimal fixture set includes:

- one reduced `playable_corpus.v1` fixture containing exactly the 4 playable items referenced by the 1 compiled/materialized question
- one `pack.compiled.v1` fixture with `question_count_requested=1` and `question_count_built=1`
- one `pack.materialization.v1` fixture with `purpose="daily_challenge"` and `question_count=1`

`assignment` materialization fixture is deferred to a later chantier if needed.

### Validation gate

Before publication, each fixture file must validate against its owner JSON schema:

- `schemas/playable_corpus_v1.schema.json`
- `schemas/pack_compiled_v1.schema.json`
- `schemas/pack_materialization_v1.schema.json`

Validation failure blocks publication and consumer mirroring.

### Versioning and drift control

- Fixture sets are immutable once published.
- New fixture semantics require a new chantier entry and explicit owner-side validation note.
- Consumer-side edits to official fixture payloads are forbidden.
