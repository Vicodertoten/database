## Standardization strategy decisions (locked)

- create pack:
  - canonical input: parameters aligned with `pack.spec.v1.parameters`
  - canonical output: `pack.spec.v1`
- list packs:
  - canonical output V1: listing view composed of `pack.spec.v1` payloads (`pack-specs` / `pack-revisions`)
- diagnose:
  - canonical output: `pack.diagnostic.v1`
- compile:
  - canonical output: `pack.compiled.v1`
- materialize:
  - canonical output: `pack.materialization.v1`
- read enrichment status:
  - currently documented as owner-side operational status flow only
  - no claim of stable public versioned contract at this stage
- request/prepare enrichment:
  - document current real inputs (`enrich-enqueue`, `enrich-execute`)
  - do not invent a new schema-versioned public payload

## Boundary rule (explicit)

`runtime-app` may orchestrate these operations later, but it must not own or redefine:

- valid parameter semantics
- compilability criteria
- diagnostics semantics
- compilation/materialization decision logic
- enrichment status semantics or queue lifecycle truth

## Risks

- over-standardizing enrichment flows that are still operational owner-side only
- inventing list/status surfaces not backed by current CLI/storage behavior
- consumer drift: runtime-app assuming semantic ownership of pack/enrichment truth

## Acceptance criteria

- owner/consumer boundary remains explicit and unchanged
- all listed operations are documented from real existing commands/artifacts
- versioned outputs are reused as canonical where they already exist
- non-versioned enrichment/status flows are explicitly documented as operational (not public stable contract)
- document is reusable by a future runtime-app orchestration chantier without redefining owner truth
